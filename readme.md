2.  What's the N+1 problem

```
N+1 means:

- 1 query to load a list
- then N extra queries for related data

Example:
- query 20 tweets
- then for each tweet query author
- then 20 more queries happen

So total becomes:
- 1 + 20 = 21 queries

How to solve
- eager load related objects with joins
- aggregate counts in SQL
- avoid querying inside Python loops
```

3.  What's the differance from cursor pagination with offset pagination.

```
Offset pagination
Example:
- page 1: `offset 0 limit 20`
- page 2: `offset 20 limit 20`

Problems
- deep offsets become slower
- unstable when new rows are inserted
- may skip or duplicate items


Cursor pagination
Use last item from previous page, for example:
- `(created_at, id)`

Then next page says:
- give me rows after this cursor in sorted order

Why better for feeds
- stable under inserts
- better performance on large data
- natural for time-ordered timeline
```

6.  why user model need this relationship, what's back_populates and cascade mean, and why need cascade.

```
    tweets = relationship(
        "Tweet", back_populates="author", cascade="all, delete-orphan"
    )
```

```
cascade="all, delete-orphan"`
This controls what happens to child objects.

`all`
Propagate ORM operations like:
- save
- merge
- delete
- refresh

`delete-orphan`
If a child object is removed from its parent relationship and has no parent, SQLAlchemy should delete it.


Why use cascade here?
The intended meaning is:

- tweets belong to a user
- if user is deleted, their tweets should usually be deleted too

Why reasonable
In this small project:
- a tweet has one author
- tweet should not exist without author

Caution
In a real production system, you may think more carefully before cascading large deletes because:
- deleting one user may delete lots of rows
- audit / compliance / soft-delete may be needed
```

7. Tweets model, why need a union index on these three column

```
__table_args__ = (
    Index("ix_tweets_user_created_id", "user_id", "created_at", "id"),
)
```

```
Because your read path often does this kind of query:

- filter by author(s)
- sort by newest first
- paginate by `(created_at, id)

The composite index matches the access pattern: find tweets by author and return them in stable time order for cursor pagination. Including `id` helps when multiple tweets share the same timestamp.
```

8. Feed model, why index on owner, not actor?

```
Because the main query pattern is:

- “show me **my** home timeline”

That means:
- filter by `owner_id`

### Definitions
- `owner_id`: the user who is reading the feed
- `actor_id`: the user who created the tweet

### Query path
Home timeline query is:
- `where owner_id = current_user`

So indexing `owner_id` is the highest-value index.

### When would `actor_id` matter?
If you often query:
- all feed rows generated by one author
- analytics by actor

Then maybe index `actor_id` too.

But for timeline serving, `owner_id` is the important one.
```

12. what is idempotent, how to make sure idempotent

```
Why important
In distributed systems:
- retries happen
- network timeouts happen
- clients resend requests

How to ensure idempotency
- unique constraints in DB
- check existing row first
- use idempotency keys for certain writes
- make delete safe if item already absent
```
